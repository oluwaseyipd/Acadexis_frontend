# Acadexis Backend Handover

This handover is built from a deep scan of the frontend codebase in `acadexis_frontend` and captures the exact backend contract expected by the client. It is intended for the Django backend developer who will implement the API and data model layer.

---

## 1. Purpose

The frontend is a Next.js 16 application using React 19, TypeScript, Zustand, Tailwind, and Axios. It currently consumes:

- `src/services/apiService.ts` — the HTTP API client with concrete endpoint expectations
- `src/services/api.ts` — a mock contract layer that mirrors additional frontend feature expectations for study sessions, courses, quizzes, notifications, and file uploads

This handover documents the backend responsibilities, endpoint shapes, model fields, auth contract, and integration notes required to make the real Django backend compatible with the existing frontend.

---

## 2. Frontend Backend Contract

### Base URL

- `NEXT_PUBLIC_API_URL` environment variable
- default fallback: `http://localhost:8000`

### Authentication

Frontend expects JWT bearer auth in `Authorization: Bearer <token>` headers.
The client stores:

- `access_token` in `localStorage`
- `refresh_token` in `localStorage`

The Axios interceptor in `src/services/api-client.ts`:

- attaches the access token to every request
- intercepts 401 errors
- calls `/auth/refresh` once using the refresh token
- retries the original request with the refreshed access token
- logs out / redirects to `/login` when refresh fails

### Role handling

Roles are enforced client-side for UX, but the backend must provide authoritative role and user claims. The supported roles are:

- `student`
- `lecturer`
- `admin` (present in app state type but not currently generated by mock data)

The backend should never rely on client role flags for security.

## 4. API Endpoint Contract

### Auth

- `POST /auth/login`
  - request: `{ email, password, rememberMe? }`
  - response: `{ user, tokens: { accessToken, refreshToken } }`

- `POST /auth/logout`
  - invalidates refresh token

- `GET /auth/google/university`
  - returns `{ url }` for OAuth redirect

- `POST /auth/google/callback`
  - request: `{ code }`
  - response: `{ user, tokens }`

- `POST /auth/forgot-password`
  - request: `{ email }`
  - response: `{ message }`

- `POST /auth/reset-password`
  - request: `{ token, newPassword }`
  - response: `{ message }`

- `POST /auth/refresh`
  - request: `{ refreshToken }`
  - response: `{ accessToken }`

### Users

- `GET /users/me`
  - returns user profile

- `PATCH /users/me`
  - partial update of profile fields

- `POST /users/me/change-password`
  - request: `{ currentPassword, newPassword }`

### Courses

- `GET /courses`
  - returns courses list

- `GET /courses/:courseId`
  - returns course detail

- `GET /modules?courseId=<id>`
  - returns modules for a course

- `GET /modules/:moduleId`
  - returns module detail

- `GET /recommendations?courseId=<id>`
  - returns recommendations for a course

### Academic Lookup

- `GET /universities`
- `GET /faculties?universityId=<id>`
- `GET /departments?facultyId=<id>`

### Course Materials

- `GET /course-materials?courseId=<id>`
  - returns uploaded materials list

- `POST /course-materials`
  - multipart/form-data
  - fields: `courseId`, `file`
  - returns processing record

### Study Lab / Chat

- `GET /chat-history?courseId=<id>`
  - returns prior chat messages for the selected course or session

- `POST /chat/send`
  - request: `{ courseId, message }`
  - returns assistant message with citations

### Notifications

- `GET /notifications`
  - returns notifications for the authenticated user

- `POST /notifications/:id/read`
  - marks notification read

### Analytics / Heatmap

- `GET /analytics/heatmap?courseId=<id>`
  - returns struggle heatmap data

### Support / Admin Requests

- `POST /support/contact`
  - request: `contact form payload`

- `POST /support/report`
  - request: `report form payload`

- `POST /support/admin-request`
  - request: `{ reason, documentProof? }`

### Quizzes

- `GET /quizzes/:courseId`
  - returns quiz metadata and questions for the course

- `POST /quizzes/:courseId/attempt`
  - not yet wired in frontend, but likely required for future quiz attempt flow

## 5. Backend Feature Requirements

### Authentication

- Use Django custom user model and JWT auth
- Implement refresh token handling and logout invalidation
- Provide user role and profile fields in the login response
- Support Google university SSO flows with a redirect URL generator and callback exchange
- Enforce university email validation in registration and login flows

### Authorization

- Students can only access their own study sessions, course history, bookmarks, and profile
- Lecturers can access course materials, uploads, analytics, student management, and admin request forms
- Use server-side role checks for every protected endpoint

### Upload / Document Ingestion

- Accept uploaded PDFs/DOCX/PPTX
- Store raw files in local dev storage or S3 in production
- Return a record with `status: processing | ready | failed`
- Launch Celery ingestion tasks that:
  - extract text & metadata
  - generate embeddings with a vector store (`pgvector`)
  - create searchable RAG documents for study lab

### Study Lab Chat

- Maintain chat history keyed to either course or study session
- Return assistant responses enriched with citation sources
- Sources must include page reference, snippet, and material name
- Support conversational context if future UI evolves beyond mock behavior

### Notifications

- Store notifications with read/unread state per user
- Provide counts and list retrieval
- Mark notifications as read via explicit endpoint
- Support `type` values used in frontend badges

### Analytics

- Provide struggle heatmap data per course
- Return average confidence as a normalized ratio (0–1)
- Include questions asked and struggling student counts

## 6. Deployment and Integration Notes

### Settings

- `AUTH_USER_MODEL = "accounts.User"`
- `CORS_ALLOWED_ORIGINS` should include the frontend origin used by the app
- `CHANNEL_LAYERS` configured for Redis for realtime notifications
- `CELERY_BROKER_URL` and `CELERY_RESULT_BACKEND` configured for Redis

### Storage

- Use local `MEDIA_ROOT` in development
- Use `django-storages` with S3 in production
- Store uploaded documents and derived assets cleanly

### Realtime

- Channels should power live notification delivery later
- The frontend currently reads notifications via polling / fetch; realtime is a future enhancement

### Testing

- Validate endpoint shapes against `src/services/apiService.ts`
- Use seed data for universities, faculties, departments, courses, and course materials
- Confirm token refresh flow and `Authorization` header handling

## 9. Notes for the Backend Developer

- There are two API contract sources in frontend: `apiService.ts` for real HTTP endpoints, and `api.ts` for mock feature behavior.
- Align actual Django serializers and model field names with the frontend data shapes.
- The frontend expects some fields to be present even if the page currently uses mocks, so implement the contracts conservatively.
- `apiService` is the authoritative HTTP client for auth and course endpoints; use it as the primary guide for endpoint naming.
- The backend should support both the academic hierarchy and the study/RAG workflows.
